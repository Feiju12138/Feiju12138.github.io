【英文】请求伪造漏洞

Posted on 2022-04-21 Edited on 2024-03-28 Views:

Introduction

The back-end of some websites can handle requests from domains other than the current domain. In this case, request forgery can be used for penetration testing.

Cross-Site Request Forgery (CSRF)

  • Cross-Site Request Forgery originates from the client-side

Exploitation

  • Suppose a website’s password modification GET request is http:example.com/reset?new_password=123456. In this case, we can create a webpage file named after the current website domain, example.com.html, and include the URI of the password modification request as a parameter for the <img> tag in the file.

The reason for renaming the file to match the current website is that some websites check if the request includes keywords related to the current website domain. Even if the forged website’s domain is different from the current website, it can still pass the validation if the website’s source code only checks for the inclusion of keywords.

More advanced requests require analysis of the structure of the original request for forgery. This is just an example.

  • As a result, the password of the user who visits this webpage will be forcefully modified.

Defense

Ineffective Defenses

  • Use Cookie for identity verification

All cookies, even encrypted ones, are submitted with each request.

  • Only accept POST requests

Not only GET requests can be forged, POST requests can also be forged by simply creating a hidden form on the webpage to construct the POST request.

  • Multi-step transactions

Multi-step transactions are not sufficient to defend against CSRF. As long as the attacker can predict every step of the entire transaction, CSRF can be achieved.

  • URL rewriting

This may be considered as a useful CSRF defense technique because attackers cannot guess the victim’s session ID. However, the user’s session ID is usually exposed in the URL, so it can still be obtained by attackers.

  • HTTPS

HTTPS itself cannot defend against CSRF, but it should be a prerequisite for any defense.

Effective Defenses

  • Validate the Referer in the request header

The Referer field in the request header is the source address of the requester. Validating the source address can effectively defend against CSRF.

  • Add token authentication

Add unforgeable information to the request.

  • Two-factor authentication

Require user to reauthenticate with password or verification code before performing critical operations, which effectively defends against CSRF.

  • Encourage users to develop good Internet habits

Do not click on links or images randomly; logout of logged-in accounts that have been idle for a long time; install appropriate security software.

Server-Side Request Forgery (SSRF)

  • Server-Side Request Forgery originates from the server-side

Code Vulnerability

  • Passing other network resource addresses as request parameters
  • The server sends a request to the URL using curl
  • Stores the data obtained from the request in a file in the current directory
  • Echoes the data obtained from the request on the webpage

Exploitation

Accessing other URLs

  • Make the server access other URLs

<url>: Network resource address

Identifying fingerprints of internal web applications
  • If the accessed URL is an internal IP address, the fingerprint of another web application in the internal network can be identified by accessing it.

<ip>: Other IP address in the server’s internal network
<webapp>: Access path of another web application

Performing port scanning

  • Use the dictionary protocol to perform port scanning, which can scan the internal network information of the server.

  • The page will only return normally when the port is open.

<port>: Port
<ip>: Other IP address in the server’s internal network

Reading server files

<src>: Local resource path of the server

Vulnerability Discovery

  • Look for functionality in web applications
    • Sharing functions
    • Transcoding functions
    • Online translation
    • Loading and downloading images
    • Bookmarking functions
    • Unpublished APIs and other URL-invoked functions
  • Look for keywords in URLs
    • share
    • wap
    • url
    • link
    • src
    • source
    • target
    • u
    • 3g
    • display
    • sourceURI
    • imageURL
    • domain

Defense Bypass

  • If the server performs string matching
    • Bypass can be achieved using the format http://example.com@real.com, where the actual accessed address is real.com
    • IP addresses can be converted to other number systems

Defense

  • Limit protocols

  • Only allow requests using the http and https protocols

  • Limit IP addresses

Avoid the application being used to obtain intranet data

  • Limit ports

Limit the requested ports to commonly used ports

  • Filter return information

  • Unified error messages

Prevent users from determining the server’s port status based on error messages. For example, return 404 for all abnormal pages.
