Preface

Nodejs Study Notes

Running JS Code in Node Environment

<file>.js: JS file to be executed

Nodejs Module Specification

  • Nodejs follows the CommonJS module specification, which defines the characteristics of modules and how they depend on each other
    • In each module, the module object represents the module itself
    • The exports property in the module object is the interface exposed to the outside
    • Executing the require() function to import objects actually loads the module.exports

Classification of Modules

  • Built-in Modules: fs, path, http
  • Custom Modules: Modules created by users
  • Third-party Modules: Custom modules shared by others

Importing Modules

Importing Built-in Modules

  • Loading built-in modules only requires specifying the module name

Importing Custom Modules

  • Loading custom modules requires specifying the module path
  • When loading custom modules, the file extension can be omitted

Importing Third-party Modules

  • Loading third-party modules only requires specifying the module name
  • When loading third-party modules, all code in the module will be executed automatically

Module Scope

  • Module scope: Variables within a module can only be accessed within the module, which helps prevent global variable pollution

Sharing Member of Module Scope Externally

  • Use the module.exports object to share members of the module scope externally. When a module is called through require(), the module.exports object is obtained
  • In a custom module, the module.exports object is initially an empty object
  • To simplify module.exports, the exports object can be used to share members of the module scope externally. Both module.exports and exports point to the same object
  • When both module.exports and exports appear, regardless of which object is assigned the final value, module.exports takes precedence

Module Loading Mechanism

  • Once a module is loaded for the first time, it is cached. This means that multiple calls to require() will not cause the code to be executed multiple times

Whether built-in, third-party, or custom modules, they will be loaded from the cache to improve loading efficiency

  • Built-in modules have the highest loading priority

  • When loading custom modules, the module must start with ./ or ../, otherwise Nodejs will attempt to load from built-in or third-party modules

  • When loading custom modules, if the extension is omitted, Nodejs will automatically complete it in the order of no extension, .js, .json, and .node. If the file is not found, an error will occur

  • When loading third-party modules, Node will concatenate directories in the order of ../node_modules/module name, ../../node_modules/module name, and so on to find the node_modules directory. If the file is not found, an error will occur

  • If a directory is passed as a parameter to the require() function as the module to load, Nodejs will first check if a package.json exists and if there is a main property

    • If there is a main property specifying the entry file, the specified entry file under the module will be loaded
    • If there is no main property specifying the entry file, the index.html file in the module root directory will be loaded
    • If none of the above finds the module’s entry file, an error will be thrown: Error: Cannot find modules 'xxx'

Npm and Packages

  • Packages are actually third-party modules
  • Packages are encapsulations by third-party developers based on built-in modules to improve development efficiency
  • Search for packages: https://www.npmjs.com
  • Download packages: https://registry.npmjs.org
  • Use Npm Package Manager to download packages

Package Management Configuration File

  • In the root directory of an Npm project, a package.json package management configuration file must be provided to record project configuration information
  • In an Npm project, the root directory name cannot contain Chinese characters or spaces, otherwise an error will be reported when creating the package management configuration file

Information Recorded

  • The package.json file records third-party modules that the project depends on through the dependencies attribute
    • Use npm install <package> to install the dependent package
  • The package.json file records third-party modules that are only used in the development phase through the devDependencies attribute
    • Use npm install --save-dev <package> to install development phase dependency packages

Creating Package Management Configuration File

  • Fill in the form as needed. If nothing is entered, it will be the default value or left blank
Quick Creation of Package Management Configuration File

Dependency Package Directory

  • When installing third-party modules, the modules will be installed in the node_modules directory
  • When importing third-party modules from the node_modules directory, simply specify the module name to import

package-lock Configuration File

  • Used to record information about the packages downloaded in the node_modules directory, including the name, version, and address of the downloaded package

Classification of Packages

  • Project Packages: Packages used only within the current project, installed in the node_modules directory
    • Development Dependency Packages: Packages recorded in devDependencies, used only during the development phase
    • Core Dependency Packages: Packages recorded in dependencies, used in both development and production phases
  • Global Packages: Usually only utility packages are installed globally
    • On Windows, they are installed in the C:\Users\Username\AppData\Roaming\npm\node_modules directory
    • On MacOS (Intel), they are installed in the /usr/local/lib/node_modules directory
    • On MacOS (Apple Silicon), they are installed in the /opt/homebrew/lib/node_modules directory

Developing Third-party Modules

Standard Package Structure

  • Each package must exist in a separate directory
  • The root directory of the package must contain a package.json configuration file. This file must include three properties: name (name), version (version), and main (entry file)
  • If a module has multiple functions, these functions can be split into separate JS files, stored in the src directory, imported into the entry file, and exposed to the outside world
